主键设计原则

  • 降低主键长度（二级索引中存放主键，防止占用太多空间，不适用UUID等）
  • 按照有序、顺序插入（自增主键）,乱序插入导致页分裂
  • 业务操作避免对主键的修改

gropup by优化

满足最左前缀原则

创建了一个包含a, b, 和 c字段的联合索引，并且运行了如下的查询：

sql
EXPLAIN SELECT b FROM table GROUP BY b;

在这个查询中，尽管你不直接访问索引的第一个字段a，但是由于你在GROUP BY子句中指定了索引的第二个字段b，MySQL可以有效地利用这个联合索引来加速分组操作。在这种情况下，MySQL能够使用索引进行查找，而不需要扫描整个表。这就是EXPLAIN输出中的Using index意味着的——MySQL正在利用索引来查找数据。

然而，即使MySQL能够利用索引来查找数据，它可能仍然需要创建一个临时表来存储分组的结果，然后对它们进行排序，以便返回正确的分组。这是因为当涉及到分组和排序操作时，MySQL通常需要确保所有的行都是按照指定的方式排序的。这就是为什么EXPLAIN输出中的Using temporary会出现的原因。

隔离等级

  • Read Uncommitted (读取未提交)
    在这种级别下，事务可以读取到其他事务尚未提交的数据变更。
    允许“脏读”(Dirty Read)，即事务 A 可以读取到事务 B 尚未提交的数据变更，如果事务 B 最终回滚，则事务 A 读取到的数据实际上是不存在的。
    这种隔离级别并发性能高，但是可能导致不一致的结果。
  • Read Committed (读取已提交)
    事务只能读取到已经提交的数据。
    解决了“脏读”的问题，但是引入了“不可重复读”(Non-repeatable Read)的问题，即在一个事务内多次读取同一条数据可能会得到不同的结果。
    这种隔离级别并发性能较高，适用于大部分场景。
  • Repeatable Read (可重读)
    事务在开始到结束期间，即使有其他事务对数据进行了修改并提交，该事务仍然可以看到事务开始时的数据状态。
    解决了“不可重复读”的问题，但是可能导致“幻读”(Phantom Read)的问题，即同一个事务内两次执行相同的查询语句，可能会出现之前不存在的新行。
    这是 MySQL 的默认事务隔离级别。
  • Serializable (可串行化)
    提供最高的数据一致性保证，所有的事务都是串行执行的。
    解决了“幻读”的问题，但是并发性能最低，因为事务之间需要严格地按顺序执行。
    在要求绝对数据一致性的场景下可能会使用。

varchar能存储多少汉字

在MySQL中，VARCHAR类型的长度限制是按照字符数来计算的，而不是字节数。这意味着，无论存储的是英文还是中文，只要字符数不超过定义的长度，就可以存储。
