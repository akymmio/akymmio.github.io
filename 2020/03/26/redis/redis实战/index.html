短信登录

集群的session共享问题

  • 为应对并发，进行水平扩展，部署多个tomcat，形成负载均衡的集群，当请求到不同的tomcat时，session会分散到不同的tomcat中，导致session不共享
  • 共享session：造成空间浪费且有延迟

session替代方案需要满足：

  • 数据共享
  • 内存存储
  • key-value形式

遇到问题：

  • 存储的数据存在long型，而stringRedisTemplate要求key-value都为String类型，所以需要进行转换

  • 手动创建的对象没有spring依赖注入，不能使用注解

优化登录拦截器

问题：当用户访问拦截器没有拦截的路径时，第一个拦截器通过token查询redis，判断存在用户，则刷新token有效期，不会拦截访问。

商户查询缓存

缓存:数据交换的缓冲区，存储数据的临时地方，读写性能好

缓存的一致性问题

主动更新实现方式：

第一种方法可控性更高（常用）
第三种方法存在可靠性、一致性问题


第一种：线程1删除缓存之后，还没来的及更新数据库，线程2未命中缓存去查数据库，查到的是没有更新的旧值,并将旧值写入了缓存，此时线程1更新完数据库，导致一致性问题

第二种：线程1未命中去查询数据库，准备更新缓存时，出现了线程2将数据库更新，并删除了缓存，此时线程1会写将旧值写入数据库

限制：库存，一人一单
使用自增ID缺点：

  • 受单表数据量的限制

全局ID

  • 唯一性
  • 高可用
  • 高性能

  • 安全性

实现：符号位0+时间戳31+序列号32

超卖



在第一个线程执行后，stock=0，而第二个线程中stock=1，不符合，实现了乐观锁
但是此乐观锁失败率高：例如大家都拿到同一个数，那么就会导致一个人成功，其他人都失败
解决：利用业务特性，下单那一刻判断库存是否>0

一人一单

方法内部加锁存在问题，会先释放锁再提交，其他线程可能进来，因为还没提交未写入数据库，新来的查询到数据库依然不存在该订单，会存在并发执行问题（事务要防止没提交就释放锁，释放锁的时机和事务是否生效）

集群部署问题
(同一个用户线程)多个jvm存在，每个jvm中都会有一个自己的锁，导致每一个锁都有一个线程获得，所以不能使用jvm内部的锁

分布式锁

分布式锁：分布式系统或集群模式下多进程可见并且互斥的锁

锁释放

锁标识：释放锁之前要判断锁标识是否一致，避免误删别人的锁

存在的问题:判断锁标识和释放锁之间阻塞，导致释放锁错误（因为已经验证过表示了，所以能删）

可以通过lua脚本，保证原子性(lua脚本下标从1开始)


释放锁脚本（获取锁标识、释放锁）

  • 锁超时释放
  • 锁释放错误

Redisson

Redission，在redis基础上实现的分布式工具的集合，包括分布式锁

setnx分布式锁不足：

  • 不可重入
  • 不可重试
  • 超时释放
  • 主从一致性

引入依赖：redission
配置redission

@Configuration
public class RedissonConfig {

    @Bean
    public RedissonClient redissonClient(){
        Config config = new Config();
        config.useSingleServer().setAddress("redis://0.0.0.0:6379");
        return Redisson.create(config);
    }
}

常用方法：

  • 获取锁：tryLock(long waitTime, long leaseTime, TimeUnit unit)
    • waitTime：等待时间，变成可重试锁，
    • leaseTime：过期时间(锁自动释放)
  • 释放锁：unlock()

Redisson可重入锁原理

锁的重入：一个线程连续多次获取同一个锁
redis的实现：核心是hash结构，以及记录获取锁的线程和重入的次数

秒杀优化

秒杀下单涉及多次的数据库操作，而数据库的并发能力比较差，耗时高


lua脚本负责保证（判断库存、是否下过单、扣减库存）操作原子性，将下单信息存入消息队列；另一个线程负责从消息队列中获取信息，操作数据库，实现异步操作

redis消息队列

优点：

  • 秒杀：抢单业务和写数据库业务分离，变成异步操作，提高并发能力
  • 数据库：减小压力


优点：

  • 利用redis存储，不受限于jvm内存上限
  • 基于redis持久化机制，数据安全性有保证
  • 满足消息队列的有序性

缺点：

  • 消息丢失：消费者消费消息失败，消息丢失（取到消息后没来的及处理，导致消息丢失）
  • 只支持单消费者

PubSub


优点

  • 发布订阅模式,支持多生产多消费

缺点：

  • 不支持数据持久化
  • 消息丢失（没人收会丢弃）
  • 消息堆积，超出时数据丢失

Stream


  • 消息可回溯（消息不消失，保存在消息队列）
  • 一个消息可以被多个消费者读取
  • 可以阻塞读取
  • 有消息漏读问题


XREADGROUP优点

  • 消息可回溯
  • 多消费者争抢消息，加快消费速度
  • 可以阻塞读取
  • 没有消息漏读
  • 有消息确认机制
